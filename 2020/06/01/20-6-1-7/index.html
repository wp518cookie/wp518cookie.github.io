6.1
网络层
一、MTU
1、不同数据链路中，MTU的大小也各不相同。当遇到需要传输数据大于MTU限制，会被分片传输。分片传输会加大路由器的负担，而且一旦某一分片异常没有到达，会导致整个IP数据报报废。
2、路径MTU
2.1、定义：
发送端主机到接收端主机之间不需要分片的最大MTU大小，即整个路由链路上中的最小MTU。然后主机根据MTU来提前分片后发送。避免在路由器上的分片处理，且可以在TCP中发送更大的包。
2.2、原理：
UDP中IP数据报设置禁止分片标志符，一旦路由器遇到需要分片才能处理的大包，不进行分片，而是将包丢弃，并返回ICMP不可大达消息将MTU值发回主机。主机根据MTU分好片后再循环之前的操作，直到没有收到任何ICMP。
TCP的情况下则是根据MTU计算出MSS长度，直接在TCP层就拆好包，避免了IP的分片。

二、IP层的协议
1、ARP：Address Resolution Protocol
通过广播根据IP获取MAC的协议
2、ICMP：Internet Control Message ProtocolAddress Resolution Protocol
检测网络是否通，例如PING命令，并告知错误原因等。

三、IP应用相关
1、DNS: (domain name system)
2、DHCP: (dynamic host configuration protocol
3、NAT: Network Address Translator，支持本地网络中使用私有地址
3.1、工作机制：
支持大量私有的Ip向公网进行通信，通过将私有Ip用公网IP+端口的方式建立映射关系，来跟公网进行通信，理论上支持65000个主机同时共用一个公有IP地址

传输层
一、socket
1、套接字的唯一定义：源IP地址+目标IP地址+协议号+源端口号+目标端口号
2、通信时端口的确认
2.1、标准既定的端口号
每个应用程序都有其指定的端口号，不能随意使用，比如http默认80，ssh默认22等
2.2、时序（动态）分配的端口号
客户端的端口可以由服务端随意分配

二、UDP
1、无连接，直接发，无法进行拥塞控制及重发及顺序保证。
2、使用场景：
2.1、包总量较少（DNS）
2.2、视屏、音频
2.3、广播通信

三、TCP
TCP通过校验和、序列号、确认应答、重发控制、连接管理及窗口控制等机制实现可靠传输
1、序列化与确认应答
数据会标明数据的序列号，应答会告知下一步接收的序列号
2、如何判断需要重发
会根据RTT时间，动态调整，到一定次数后便会放弃重发
3、连接建立：
syn + seqA——> ack (seqA + 1) syn (seqB) -> ack(seqB)
只有两次的话，当客户端的syn包延迟到达了，以至于连接都已经建立并关闭或者ack包迟迟未到客户端，服务端却以为连接已经建立了，一直等待着。
4、四次挥手：
如何判断数据已经发完了，得有个机制，就是FIN机制，表明已经没有数据发送了。
FIN + seqA ——> ack (seqA + 1) , Fin(seqB) ——> ack(seqB + 1)
5、MSS:maximum segment size， 最大消息的长度
最理想的情况是长度刚好是IP中不会被分片的最大数据长度

6.2
一、TCP
1、利用窗口控制提高速度
这个MSS值在sync时会带上，避免IP触发分片。
为了避免干等上个应答，TCP引入了窗口概念。窗口大小就是说无需等待确认应答而可以继续发送数据的最大值。
2、重发控制
2.1超时重传机制：
根据RTT估算出超时的时间，来重传
2.2快速重传机制：
当连续3次收到同一个序列号的应答，则会对该数据进行重发，比超时管理更高效。
3、流量控制
TCP首部中会有一个字段用来通知发送方自己的缓冲区大小，避免溢出。当缓冲区慢了，tcp会返回报错。
4、拥塞控制
TCP慢启动机制，防止一上来发送大量数据，导致网络拥堵。慢启动时，拥塞窗口的大小设置为1MSS。

二、HTTP
1、http1.0,1.1,2.0
http1.0:每个http请求建立的tcp链接无法复用，需要频繁的三次握手
http1.1:
1.1、支持长连接，支持一个连接中进行多次http请求，不需要等待上一次请求结果返回即可发出下个请求
1.2、新增了HOST首部，支持在同一台IP的机器上部署多个app，通过host来区分到底访问哪个网站

三、https(http + ssl)
1、SSL(Secure Socket Layer)
http是http ——> tcp ，https是http ——> ssl ——> TCP
2、加密算法
2.1、对称加密算法：加密解密用的是用一个key
例如：DES，AES
优点：速度块，加密效率高
缺点：密钥如果要唯一，量很大。同时密钥一旦泄露，加密信息不安全。特别网络请求，怎么把密钥给别人？所以对称加密就很扯淡。
2.2、非对称加密算法
例如：RSA
一对密钥由公钥和私钥组成。公钥能解密私钥加密的数据，反之亦然。
优点：安全
缺点：效率低
2.3、SSL连接建立过程
连接还是照常建立，连接建立后，进行下述操作。
1、客户端发起Client Hello开始SSL通信，报文中包含客户端支持的SSL指定版本，加密组件等。
2、服务端收到客户端SSL请求后，会发送客户端支持的SSL版本及加密组件。
3、服务端发送公钥
4、服务端告知Server Hello Done报文通知客户端,最初阶段的SSL握手协商部分结束。
5、SSL第一次握手结束后，客户端验证完公钥后，客户端以Client Key Exchange报文作为回应，同时会带上公钥加密的随机密码串。
6、客户端继续发送Change Cipher Spec报文，提示服务器之后通信用之前的密码串。
7、客户端发送全部报文的整体校验值。
8、服务端同样发送Change Cipher Spec报文
9、服务端同样发送Finish报文
10、报文交换完毕后，SSL才算完成。

解释： 当前时间 |系统运行时间 | 当前登陆用户 | 最近1分钟、5分钟、15分钟系统平均负载
平均负载：系统处于可运行状态和不可中断状态的平均进程数，即平均活跃进程数。
可运行状态： 正在使用CPU或者正在等待CPU的进程，即PS命令时，处于R状态（Running或Runnable)的进程。
不可中断状态：正处于内核态关键流程中的进程，且这些流程不可打断，如等待硬件设备的IO响应。如向磁盘写数据，为了保证一致性，在得到磁盘回复前不允许被其他进程中断。

6.3
一、JIT
1、什么情况下会触发JIT
方法调用计数器和循环回边计数器达到某阈值后，会进入编译队列，队列由一个或者多个后台线程来异步处理。
-XX:CICompilerCount=N设置JVM队列处理线程总数，c1和c2线程数量比为1:2。
2、方法内联
是JIT编译在运行时完成，当方法的调用次数达到一定阈值的方法调用替换为方法体本身
2.1、限制条件：
1）调用次数是否达标
2）被调用方的方法大小是否合适
3）被调用方运行时其实现是否可以唯一确定
2.2、针对虚方法的去虚化
1）完全去虚化：通过类型推导或者类层次分析。
2）条件去虚化：直接把class判断展开写死在代码里。
3、逃逸分析：判断新建的对象是否会被其他线程访问到
开启逃逸分析后，会采取更激进的优化措施。
